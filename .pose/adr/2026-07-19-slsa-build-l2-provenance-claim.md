# ADR: SLSA Build L2 provenance claim

## Status
Accepted (2026-07-19) — spec `pose-slsa-provenance`

## Context

Signatures (previous milestone) prove who published an artifact; they do not
prove how it was built, from which source revision, by which builder. The
spec requires provenance linking artifacts to source, builder and invocation
— with the explicit constraint of claiming only what the build model
actually provides ([SLSA 1.2](https://slsa.dev/spec/v1.2/)).

Builder threat model: GitHub-hosted runners execute the release workflow.
The build is ephemeral and credential-scoped (OIDC, no long-lived keys), but
it is **not** isolated from the repository's own workflow definition — a
maintainer (or compromised dependency of the workflow) can alter the build
process. That is exactly the boundary between SLSA Build L2 and L3.

Alternatives considered:

1. **slsa-github-generator reusable builder** — supports a Build L3 claim
   (isolated, externally defined builder), but constrains the pipeline shape
   and adds a trusted third-party workflow; deferred until the trust story
   demands L3.
2. **GitHub artifact attestations** (`actions/attest-build-provenance`) —
   first-party, generates signed SLSA v1 provenance with subjects by digest,
   verifiable by any consumer with `gh attestation verify`; supports an
   honest Build L2 claim today.

## Decision

Option 2, with an explicit, bounded claim:

- **Subjects (R1):** every release archive and `checksums.txt`, identified
  by SHA-256 digest, attested in the release workflow after GoReleaser
  produces `dist-release/`.
- **Contents (R2):** the SLSA v1 predicate records source repository and
  revision, builder identity (the release workflow ref) and invocation
  metadata. No secrets enter the predicate; signing uses the ephemeral OIDC
  identity (`attestations: write` + `id-token: write`, both already scoped).
- **Claim:** **SLSA Build L2** — hosted build platform, signed provenance
  generated by the build platform. **We do not claim L3**: the build is not
  isolated from the workflow definition. This limitation is published in
  `SECURITY.md` beside the claim.
- **Verification (R3):** `gh attestation verify <artifact> --repo
  oseiaspereira88/pose --signer-workflow
  oseiaspereira88/pose/.github/workflows/release.yml` rejects modified
  artifacts (digest mismatch), wrong repositories and untrusted builders.
  The independent verifier workflow runs this for the platform archive and
  the checksum manifest on every published release.
- **Contract test:** `TestAttestedReleaseContract` pins the attestation
  step, its subjects and the permission scopes.

## Consequences

- Positive: consumers can tie every artifact to a source revision and
  builder with a standard in-toto/SLSA predicate and stock tooling.
- Positive: no release-process mutation is manual — provenance generation is
  part of the tag pipeline.
- Trade-off: provenance verification requires `gh` and GitHub API
  availability (the attestation store), unlike the offline Sigstore
  bundles; the two layers compose rather than replace each other.
- Residual: a signed but weakly isolated build can overstate the guarantee —
  mitigated by claiming exactly L2 and documenting the L3 gap; upgrading to
  an isolated builder is future roadmap work if demanded.
